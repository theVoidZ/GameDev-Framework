Detailed Description

The HierarchicalContainer class provides a specific way of orgnazing components.

It is structured in a multi-root tree. A non cyclic directed graph with several source-nodes and several (final/well?) nodes.

Every node can have multiple parents and multiple chidren. Note that, the top-most nodes have no parents, and the bottom-most nodes have no children. See Component

Here as shown in the figure, all_items container contains all the components regardless of their positionning in the hierarchy.

As for the top_most_items container, it contains only Layer 1's components. ex: n1, n2, n3

This tree structure allows to create a dependency relationship between components. For example, n5 depends on n1 and n2.

template<typename T >
T* gdf::kernel::HierarchicalContainer::addComponent ( std::list< T * > *  container = nullptr)

Add a newly created component of type T.

When ever a component is added, all_items keeps trace of it. Besides, if the component does not depend on other components, it will be added on Layer 1. i.e in top_most_items.

Otherwise, it is added as a child of the components it depends on.

The container argument when specified, allows to automatically insert that created component into this container. Otherwise, it does nothing.

for example: In order to keep track of all cameras within the Scene. Each time the scene adds a camera component, it will keep track of it inside a container called 'cameras'

scene->addComponent<Camera>( &scene->camera );

Otherwise, the method is simply called so

scene->addComponent<GarbageCollector>();
Note
This method always return a new component, never returns nullptr.
Parameters
containerThe container where this component is optionally added into it.
Returns
A new component.

Checks whether a component comp can be attached to the hierarchy of components.

Based on a pre-defined rules in KernelRules. A component can be attached if it satisfies on of those rules.

A rule is defined as follow:
\( R â†’ C1, C2, ....Cn \), where R, C1, C2,..., Cn are class names

A rule is statisfied toward an object Obj if and only if:

  • Each item of the second part of the rule must have an instance (sub_class) within the target object.
  • If so, Then the Component A can be attached to the object Obj.
Parameters
compComponent to test its attachement with the current Object
lcOutput variable to store the involved components during the checking. This is used in combination with make_linking.
Returns
True if the component can be attached, false otherwise.
