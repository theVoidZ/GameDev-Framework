Detailed Description

The KernelRules class defines a set of rules in order to check the satisfiability of the add and the remove actions.

It is mainly used with HierarchicalContainer::addComponent and GameObject::destroy in order to check whether these actions can be performed.

Since, the HierarchicalContainer defines a dependency relationship between components, whenever addComponent is invoked on T ( template method ). A checking process must be performed in order to verify the satisfiability of the 'T rule'.

Suppose we have:

//kernelrules
create_rule("TimeWinder", "Chrono");
create_rule("TimeWinder", "Transform");
create_rule("TimeWinder", "TimeKeeper");
create_rule("Renderer", "Transform");

This generates the following rules: ( â†’ means: requires )

\( Renderer â†’ Transform \)

\( TimeWinder â†’ Transform, TimeKeeper, Chrono \)

Thus, in order to add a TimeWinder Component on a given GameObject, the game object must already have all the listed (and required) components of the "TimeWinder" rule.

Otherwise, the component will not be created

Same case for the destroy action, but condition check is applied in a reverse way. Means that you cannot remove a component if another active component depends on it.

Multiplicity

Another rule type is about instance multiplicity.

These set of rules allow a limited number of T-type Component within the same GameObject/HierarchicalContainer.

//kernelrules
make_singleton("Transform");
//GameObject
addComponent<Transform>(); // Transform component is created
addComponent<Transform>(); // Error . a transform already exists, as Transform is marked as singleton ( x1 )

User-defined-rules

Another point is about defining User-rules

It allows the user to define its own set of rule according to its application logic.

With a (dummy) example:

Note
A hero represents a scene object (GameObject). and weapon equippements are Components
  • A hero cannot have more than one sword and one shield.
  • A sword cannot be equipped unless the hero is equipped with combat-gloves.

The generated rules whould be:

// user_kernelrules
make_singleton("Sword");
make_singleton("Shield");
create_rule("Sword", "CombatGloves");

Note that CombatGloves is not affected by the singleton rules, means that there is no number restriction for that type of object.
